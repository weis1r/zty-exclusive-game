### 2. 经典玩法回收与数值重整

#### 主要实现
- 将目标玩法收敛回经典玩法。
- 每关块数调整为更平滑的递进曲线：
  - `36 / 48 / 60 / 72 / 84`
  - 第 `5` 关后封顶 `84`
- 动态块规则继续保留，但围绕经典盘面做节奏收口。
- 局内加入倒计时、提示、撤销等玩法辅助。

#### 实现思路
- 不再追求过于复杂的机制叠加，而是让玩家优先看懂局面。
- 用块数、倒计时、提示次数控制前中期节奏。
- 保留动态块机制作为变化来源，但不让它盖过基本配对体验。

#### 遇到的问题
- 之前局内信息过多，玩家视线容易被 HUD、提示文本、装饰元素干扰。
- 数值和提示逻辑同时存在时，很容易出现“看起来热闹，但操作反馈并不清晰”的问题。

#### 解决方案
- 局内 HUD 收口为更少的信息项。
- 反复减少视觉噪音，只保留关卡、剩余块数、倒计时等核心信息。
- 用更直接的“消除”替代弱提示，强化道具反馈。

### 3. 棋盘视觉降噪与单层棋盘修复

#### 主要实现
- 修复“棋盘像有两层”的问题。
- 最终确认问题不在 DOM 结构本身，而在 `.game-board::before` 伪元素额外画了一层内框。
- 去掉第二层伪元素后，棋盘恢复为真正的单层表现。
- 继续压低局内噪音，包括：
  - 顶部信息块
  - 托盘和底栏厚重感
  - 牌面立体感
  - 动态块小气泡的存在感

#### 实现思路
- 先确认“问题是结构还是视觉假象”，避免误删布局层。
- 把棋盘、牌面、HUD 看成同一张视觉画布，优先让棋盘完整、平整、可读。
- 优先减少“框中有框、卡上有卡”的堆叠感。

#### 遇到的问题
- 用户反馈“还有两个图层”，初看容易误判为页面里真的有双容器。
- 实际上是视觉伪元素造成的误解。

#### 解决方案
- 先排 DOM，再排 CSS 伪元素。
- 删除 `.game-board::before` 的额外内框。
- 随后把局内其他视觉元素也做了一轮压平，保证棋盘成为唯一视觉中心。

### 4. Android 兼容性问题排查与修复

#### 主要实现
- 修复 Android 打包后首页 / 单局页只显示上半屏、下半屏发黑的问题。
- 修复 Android 包内“上一步 / 撤销”在返回键下不生效的问题。

#### 实现思路
- 不猜原因，直接在 MuMu 模拟器里复现：
  - 安装 APK
  - 截图
  - 看前台 Activity
  - 抓 `logcat`
- 先分清是：
  - App 没启动
  - WebView 白屏
  - 页面只渲染半屏
  - 返回键被系统层或 WebView 吞掉

#### 遇到的问题 A：半屏显示 / 下半屏黑色
- MuMu 中首页不是白屏，而是内容只渲染到上半屏。
- 根因是 Android WebView 下 `vh / dvh / 100%` 高度链路不稳定，容器没有真正撑满。

#### 解决方案 A
- 在前端增加 `window.innerHeight -> --app-height` 的同步机制。
- 让 `html / body / #root / app-shell / device-shell` 全链路用 `--app-height` 兜底。
- 同时让屏幕主容器切到更稳定的 `flex` 结构。

#### 遇到的问题 B：安卓返回键没有触发局内撤销
- 浏览器中按钮逻辑正常，但打包成 Android 后，按返回键会直接走 WebView 历史后退或回首页。
- 根因是 `WebView` 聚焦时会优先消费 `KEYCODE_BACK`，前端状态机没有收到返回事件。

#### 解决方案 B
- 前端暴露 `window.androidHandleBack()`：
  - 设置打开时先关闭设置
  - 单局优先执行撤销
  - 结果页执行页面内自己的返回逻辑
- Android `MainActivity` 同时接管：
  - `dispatchKeyEvent(KEYCODE_BACK)`
  - `OnBackPressedDispatcher`
- 原则改为：
  - 先问前端是否处理
  - 前端处理了就不再走系统回退
  - 前端未处理，才继续走 WebView 或系统返回

#### 验证方式
- MuMu 安装并启动 APK
- 使用 `adb` 截图对比修复前后
- 抓 `logcat`
- 实际在局内点入一张牌后按安卓返回键，观察托盘是否撤回

#### 验证结果
- 页面已能铺满整屏
- 单局页正常显示
- 安卓返回键已能触发局内撤销

- `src/main.tsx`
- `src/index.css`
- `src/App.tsx`
- `android/app/src/main/java/com/zty/exclusivegame/MainActivity.java`

### 5. 单局手感继续优化

#### 主要实现
- 放大单局牌面点击区域，面积约提升 `50%`
- 修复残局自动通关条件：
  - 不再只看“棋盘剩余块数”
  - 改为看“全局未解决块数”
  - 计入棋盘、托盘、暂存位
  - 当总数 `<= 4` 时直接判胜

#### 实现思路
- “点起来太小”本质上不是单纯的视觉问题，而是命中面积问题。
- “剩 3 块没通关”不是单纯 UI 不刷新，而是胜利判定口径太窄。
- 所以一边放大牌面命中框，一边把残局判定从“局部剩余”改成“全局剩余”。

#### 遇到的问题
- 不能只把牌放大，否则位置会漂、层叠关系会乱。
- 不能只看棋盘剩余块数，否则托盘里挂着 1 张时会出现用户感知和逻辑不一致。

#### 解决方案
- 以牌中心点为基准扩张牌面尺寸，而不是简单向右下拉大。
- 胜利判定改成：
  - `remainingBoardTileCount`
  - `trayTiles.length`
  - `orbitPockets` 中非空数量
  - 三者相加后统一判断
- 增加引擎级回归测试，固化“全局剩 3 块也能自动通关”的场景。

#### 关键文件
- `src/screens/GameScreen.tsx`
- `src/App.css`
- `src/game/engine.ts`
- `src/game/engine.test.ts`

### 6. Android 返回按钮确认框修复

#### 主要实现
- 修复单局左上角返回按钮“点击后没反应”的问题。
- Android 包内现在会正常弹出“确定要放弃这一局并返回首页吗？”确认框。

#### 实现思路
- 先确认按钮本身是否点中、事件是否触发，而不是直接怀疑布局遮挡。
- 再排查这条点击链路后续依赖的浏览器能力：
  - `onClick`
  - `handleBackFromGame`
  - `window.confirm`
  - Android WebView 对 JS 弹窗的支持

#### 遇到的问题
- 前端逻辑本身没有问题，按钮点中后确实会进入 `handleBackFromGame()`。
- 但 Android WebView 没有正确接管 `window.confirm(...)`，导致确认框不弹。
- 用户看到的现象就是按钮“没反应”。

#### 解决方案
- 在 Android 壳层新增 `GameWebChromeClient`。
- 使用原生 `AlertDialog` 显式接管 `onJsConfirm(...)`。
- `MainActivity` 中为 `WebView` 挂载新的 `GameWebChromeClient`，不再依赖默认实现。

#### 验证方式
- 重打 Android Debug 包并安装到 MuMu。
- 启动应用，进入单局。
- 点击左上角返回按钮。
- 实际观察是否弹出确认框。

#### 验证结果
- MuMu 中已经可以稳定弹出确认框：
  - 文案：`确定要放弃这一局并返回首页吗？`
  - 按钮：`CANCEL / OK`

#### 关键文件
- `android/app/src/main/java/com/zty/exclusivegame/GameWebChromeClient.java`
- `android/app/src/main/java/com/zty/exclusivegame/MainActivity.java`

## 两天里反复出现的共性问题

### 1. 视觉问题未必是结构问题
- 典型案例：双层棋盘
- 经验：先看 DOM，再看伪元素、阴影、边框和背景叠层

### 2. Android 问题未必是“白屏”
- 典型案例：实际上是半屏渲染
- 经验：先截图，再测 Activity，再看日志，避免误判

### 3. 用户体感问题常常是“逻辑 + 表现”叠加
- 典型案例：
  - 提示不明显
  - 牌太小
  - 剩 3 块没通关
- 经验：不要只改样式或只改状态机，要同时看玩家实际看到的结果
